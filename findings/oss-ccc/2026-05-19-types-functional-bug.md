### Bug: `align_up` — returns misaligned offset on overflow

```rust
pub fn align_up(offset: usize, align: usize) -> usize {
    if align == 0 { return offset; }
    let mask = align - 1;
    match offset.checked_add(mask) {
        Some(v) => v & !mask,
        None => offset, // overflow: return offset unchanged   <-- BUG
    }
}
```

The function uses `checked_add` to *prevent* the integer overflow
panic on `offset + mask`. But when overflow would occur, it returns
`offset` **unchanged** — which may not be aligned to `align`.

Example: `align_up(usize::MAX, 8)` returns `usize::MAX`. But
`usize::MAX % 8 = 7 ≠ 0`. The function's contract (silently — there's
no doc comment) is "result is a multiple of `align`", and this branch
violates it.

The defensive spec produces VERIFIED-clean for this function: no
panic, no slice OOB, no overflow (because `checked_add` is correct
about not crashing). The **Phase 1 functional spec** generated by the
LLM was:

```
precondition: (align == 0) || ((align & (align - 1)) == 0)
postcondition: ((result >= offset) && ((align == 0) || (result % align == 0)))
              && ((align == 0 && result == offset)
                  || (align != 0 && result >= offset && result % align == 0
                      && result - offset < align))
```

Kani found a CEx satisfying the precondition (`align` is power of 2)
where the postcondition is violated — exactly the `None => offset`
branch.

## Why this matters

This is the **first** Phase 1 functional-spec find in CCC that was a
real bug rather than a Phase 1 false positive. It's the bug class
Phase 1 was built for: silent semantic violations that don't manifest
as panics.

The defensive spec misses bugs in this class entirely (`checked_add`
makes the body panic-free). Only by comparing the implementation to
a reference behaviour does the discrepancy show up.

## Impact

`align_up` is called from struct-layout code (`types.rs` is the type-
system module). A struct with a field at offset = `usize::MAX` would
trigger this — practically unreachable, but the contract violation
is silent in release builds and would produce subtly wrong memory
layout if a caller didn't pre-validate `offset`.

## Fix sketch

```rust
pub fn align_up(offset: usize, align: usize) -> Result<usize, LayoutError> {
    if align == 0 { return Ok(offset); }
    let mask = align - 1;
    offset
        .checked_add(mask)
        .map(|v| v & !mask)
        .ok_or(LayoutError::OffsetOverflow)
}
```

Surface the overflow case to the caller as an error instead of
silently returning an out-of-spec value.

## Classification

REAL_BUG under any threat model. The implementation violates its own
implicit contract; no caller can assume the function honours
alignment if `offset + align - 1` could overflow.
