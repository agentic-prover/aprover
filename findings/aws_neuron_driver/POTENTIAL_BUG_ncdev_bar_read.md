# Potential heap-OOB-read in ncdev_bar_read (AWS Neuron driver)

**Status:** UNCONFIRMED. Discovered by bmc-agent during automated
sweep. Requires manual verification + reproducer construction before
reporting to AWS.

**Target:** `aws-neuron-driver/neuron_cdev.c` (latest main as of
2026-05-21 clone)
**Function:** `ncdev_bar_read` (line 1478)
**Bug class:** Kernel heap out-of-bounds read via IOCTL
**Severity (tentative):** Local kernel info-disclosure. Requires
`/dev/neuronN` open access (typically root or `neuron` group).
Triggered through `NEURON_IOCTL_BAR` (BAR_READ).

## bmc-agent finding

CBMC verdict:
```
[ncdev_bar_read.pointer_dereference.11] line ... dereference failure:
    pointer outside object bounds in reg_addresses[(signed long int)i]
```

## Root cause

Two-function call sequence with mismatched array sizing:

```c
// Caller (ncdev_bar_rw at neuron_cdev.c:1622):
static long ncdev_bar_rw(struct neuron_device *nd, void *param, bool read)
{
    struct neuron_ioctl_bar_rw arg;
    u64 *reg_addresses = NULL;
    u64 address_count;

    neuron_copy_from_user(__func__, &arg, ..., sizeof(arg));   // arg from user

    if (arg.bar == 0)
        address_count = arg.count;
    else
        address_count = 1;                                     // !!! one element
                                                                //     when bar != 0

    reg_addresses = kmalloc(address_count * sizeof(u64), GFP_KERNEL);
    neuron_copy_from_user(__func__, reg_addresses, arg.address,
                          address_count * sizeof(u64));

    if (read)
        ret = ncdev_bar_read(nd, arg.bar, reg_addresses,
                             arg.data, arg.count);             // !!! arg.count passed,
                                                                //     not address_count
```

```c
// Callee (ncdev_bar_read at neuron_cdev.c:1478):
static long ncdev_bar_read(struct neuron_device *nd, u8 bar,
                           u64 *reg_addresses, void *user_va,
                           u32 data_count)
{
    ...
    if (bar == 0 || bar == 2) {
        ...
        for (i = 0; i < data_count; i++) {                     // !!! reads reg_addresses[i]
            u64 bar_addr = (u64)nd->npdev.bar0;                //     for i = 0..data_count-1
            u64 bar_size = nd->npdev.bar0_size;

            if ((reg_addresses[i] < bar_addr) ||
                (reg_addresses[i] >= bar_addr + bar_size)) {
                return -EINVAL;
            }
        }

        data = kmalloc(data_size, GFP_KERNEL);
        ret = ndhal->ndhal_fw_io.fw_io_read_csr_array(
            (void **)reg_addresses, data, data_count, true);   // ! re-uses reg_addresses
        ...
    }
    ...
}
```

## Exploit conditions

1. `arg.bar == 2` (passes the validation gate `bar == 0 || bar == 2`,
   but takes the `address_count = 1` branch in the caller).
2. `arg.count > 1` (large enough to read past the 8-byte allocation).
3. Sufficient OOB-read bytes fall in the bar0 range to bypass
   validation, OR the OOB-read causes a kernel page fault.

## Impact

Each iteration of the validation loop reads `reg_addresses[i]` for
`i = 0 .. arg.count - 1`. The buffer is sized for ONE element only.
This is a **heap OOB read** in kernel space.

Even if the validation gate stops the function from proceeding to
`fw_io_read_csr_array`, the OOB-read itself is the bug class:
- Adjacent kernel heap memory is read
- Side-channel observable (e.g. via timing of `-EINVAL` return vs
  proceeding to the kmalloc on the next line)
- If the OOB bytes happen to satisfy the bar range check, the
  function proceeds and `fw_io_read_csr_array` reads `arg.count`
  registers from the OOB pointer array — leading to MMIO reads from
  arbitrary addresses controlled by adjacent heap contents.

## Reachability

`/dev/neuronN` permissions vary by deployment:
- AWS-provided EC2 Trainium/Inferentia AMIs: typically root-only
- Custom setups with `neuron` group: any user in that group

So exploitability depends on deployment, but the bug class is real
regardless: any process with /dev/neuronN access can trigger.

## Suggested fix

Either:
1. Pass `address_count` (not `arg.count`) to ncdev_bar_read/write.
2. Bound `arg.count <= address_count` in the caller before the call.
3. Recompute `address_count` from `bar` inside ncdev_bar_read and
   bound the loop accordingly.

The minimum fix is to add a length check before the loop:
```c
if (bar != 0 && data_count > 1) {
    return -EINVAL;  // BAR2 only supports single-address reads
}
```

The comment in the source says "BAR2 reads are always sequential and
so addresses are autogenerated from base" — but the validation loop
reads addresses[i] anyway, which contradicts the autogeneration intent.
The fix above aligns the code with the comment.

## What bmc-agent's verdict means

CBMC's `pointer_dereference.11` failure shows: there is a path through
the function where `reg_addresses[i]` is dereferenced for `i` larger
than the buffer size, AND the verification harness's allocation makes
that buffer at least 5 elements. Without inter-function reasoning,
CBMC doesn't know `reg_addresses` was sized to 1 element by the
caller — but the bug exists in the source regardless of CBMC's
modelling of it. The CBMC verdict is what prompted the analysis;
the bug is in the call-sequence math, not in CBMC's specific
verdict.

**Until a KASAN-confirmed PoC is constructed, this remains
"discovered but unconfirmed".** Do not file public disclosure.
