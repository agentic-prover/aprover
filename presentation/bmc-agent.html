One agent, three moves

A closed loop that turns "looks fine" into a checkable claim — and keeps refining until the verdict is solid.

✦
01 · synthesize

LLM spec synthesis

An LLM reads the code and proposes the pre/post-conditions it must obey — the contract no one wrote down.

⚙
02 · check

Bounded model checking

CBMC / Kani exhaustively explore bounded executions, hunting an input that violates the spec.

⚖
03 · classify

Classify the counterexample

Real bug or false alarm? If the spec was wrong, CEGAR refines it and re-checks. If the bug is real, replay it under ASan.

You can't just check each part

At scale the system is thousands of functions that call into each other. The dangerous bug usually isn't inside any single component — it emerges along a path that threads through several, and your tests may never walk that path.

That's why per-component review and unit tests miss it. bmc-agent reasons over the composition and pins the bug to a concrete, reachable trace.

parse_chunk→ read_tag→ buf[idx] spec idx < len violated · counterexample len=8, idx=12 · replayed under ASan → SEGV
